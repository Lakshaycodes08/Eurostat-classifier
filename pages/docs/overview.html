Overview

Swytchcode treats AI as a planner, not an executor. Only swytchcode exec runs tools; editors and agents call it – they don’t run arbitrary code.

Core concepts

Execution authority

All tool runs go through swytchcode exec. No other code path executes methods or workflows. Agents and editors are read-only over the tool list; they request execution, they don’t perform it.

tooling.json (the contract)

Single source of truth for what is allowed. Human-readable, machine-enforced. Only tools listed in tooling.json can be executed. No implicit allowlists.

Wrekenfile (what is possible)

Defines METHODS, WORKFLOWS, and STRUCTS for an integration. Fetched with swytchcode get / bootstrap. Describes capabilities; tooling.json decides which of those are enabled for this project.

Canonical ID

Stable identifier for a method or workflow (e.g. api.cluster.create). Used in add, list, info, and exec.

Methods vs workflows

Methods = single callable. Workflows = multi-step. Both are tools and both go through exec.

Deterministic JSON

Exec input and output are JSON. No prompt-based execution. Failures are explicit (exit codes, structured errors). No hidden retries or agent-side interpretation of policy.

Registry vs local

swytchcode search hits the remote registry. get, bootstrap, list, and exec use local data only – no runtime registry calls.

Typical flow

  1. swytchcode init – create .swytchcode/ and tooling.json
  2. swytchcode get <project> – fetch an integration bundle
  3. swytchcode add <canonical_id> – add a tool to tooling.json
  4. swytchcode exec <canonical_id> – run the tool (from CLI or JSON stdin)
